## Overview

Moss SDK can be configured in multiple ways to fit your application's needs. This guide covers all configuration methods:

1. **Dashboard Settings** - Web UI for managing application-level settings
2. **SDK Initialization** - JavaScript configuration when loading the SDK
3. **Runtime API** - Dynamic configuration changes via JavaScript

<Tip>
    Dashboard settings provide defaults that can be overridden by SDK
    initialization options. Runtime API calls take precedence over both.
</Tip>

## Configuration Priority

When the same setting is configured in multiple places, Moss uses this priority order (highest to lowest):

1. **Runtime API calls** - `window.Moss.configure()`
2. **SDK initialization parameters** - Script tag or constructor options
3. **Dashboard settings** - Application-level defaults

<Info>
    This allows you to set sensible defaults in the dashboard while giving
    developers fine-grained control per user or session.
</Info>

## Application Information

View and manage your application's metadata and identity settings in your dashboard.

### Application Name

Your application's display name in the dashboard. This field is currently read-only.

### Application ID

Your unique application identifier. Use this when integrating the SDK.

### Created Date

When your application was created.

### UI Language

Controls the language displayed to end users in the SDK interface.

**Default:** Auto (detects user's browser language)

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > Application Information** and select from the Language dropdown.
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      language: 'en' // 'en', 'ko', or undefined for auto
    })
    ```
  </Tab>
  <Tab title="Runtime">
    ```javascript
    window.Moss.configure({ language: 'ko' })
    ```
  </Tab>
  <Tab title="URL Parameter">
    ```html
    <script src="...?language=en"></script>
    ```
  </Tab>
</Tabs>

<Info>
    When set to "Auto" (or undefined in SDK), Moss will automatically detect and
    use the user's preferred browser language.
</Info>

## General Settings

Configure your application's core AI assistant features.

### Enable Vision

Allow the SDK to process and analyze screenshots and visual content. When enabled, the AI assistant can "see" the user's screen and provide more contextual guidance.

**Default:** `true`

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > General Settings > Core Features** and toggle "Enable Vision".
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      useVision: true // or false to disable
    })
    ```
  </Tab>
  <Tab title="Runtime">
    ```javascript
    // Vision setting is typically set at application level
    // Contact support if you need runtime control
    ```
  </Tab>
</Tabs>

**Use cases:**

-   Visual troubleshooting
-   UI element identification
-   Screenshot-based guidance

<Tip>
    Enable this feature if your users need help finding specific UI elements or
    if you want the AI to provide visual context.
</Tip>

### Enable Session Replay

Record and upload user sessions for replay in the dashboard. This allows you to review user interactions and debug issues.

**Default:** `true`

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > General Settings > Core Features** and toggle "Enable Session Replay".
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      enableSessionRecording: true
    })
    ```
  </Tab>
  <Tab title="Runtime">
    ```javascript
    window.Moss.configure({ 
      enableSessionRecording: false // Disable for privacy-sensitive users
    })
    ```
  </Tab>
</Tabs>

**What's recorded:**

-   User interactions (clicks, inputs, navigation)
-   DOM structure changes
-   Timestamps and session metadata

<Warning>
    Session recordings may contain sensitive user data. Ensure you have
    appropriate privacy policies and user consent.
</Warning>

### Enable Screen History for AI

Include user action history (clicks, inputs, navigation) and DOM changes in the AI's context. This provides the AI with a richer understanding of the user's journey.

**Default:** `true`

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > General Settings > Core Features** and toggle "Enable Screen History for AI".
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      enableScreenHistoryRecording: true
    })
    ```
  </Tab>
  <Tab title="Runtime">
    ```javascript
    window.Moss.configure({ 
      enableScreenHistoryRecording: true 
    })
    ```
  </Tab>
</Tabs>

**Benefits:**

-   More contextual AI responses
-   Better understanding of user intent
-   Improved multi-step guidance

### AI Assistant Instructions

Provide custom instructions to help the AI assistant navigate your application or provide context not available through documentation.

**Default:** Empty  
**Maximum:** 30,000 characters

<Note>
    **Configuration:** Dashboard and API only (not available in SDK
    initialization)
</Note>

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > General Settings > AI Assistant Instructions** and enter your custom instructions in the textarea.
  </Tab>
  <Tab title="API">
    ```javascript
    // Use the API client to update instructions programmatically
    fetch(`/api/applications/${applicationId}/instructions`, {
      method: 'PUT',
      headers: { 
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${apiKey}`
      },
      body: JSON.stringify({
        instructions: 'Your custom instructions here...'
      })
    })
    ```
  </Tab>
</Tabs>

**What to include:**

-   Specific navigation patterns unique to your application
-   Important UI context or workflow information
-   Special considerations or edge cases
-   Terminology or concepts specific to your domain

**Example:**

```
When users ask about 'dashboard', always guide them to the Analytics
section first, not the Overview. The main navigation menu collapses
on mobile devices and requires clicking the hamburger icon in the
top-left corner.
```

<Tip>
    Use clear, concise language. The AI will incorporate these instructions when
    helping users.
</Tip>

## User Experience

Configure planning features and session behavior.

### Planning Configuration

Enable AI-powered multi-step planning for complex user tasks.

#### Enable Planning

Turn on/off the planning feature entirely.

**Default:** `false`

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > User Experience > Planning Configuration** and toggle "Enable Planning".
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      planning: {
        enabled: true
      }
    })
    ```
  </Tab>
  <Tab title="Runtime">
    ```javascript
    window.Moss.configure({
      planning: { enabled: true }
    })
    ```
  </Tab>
</Tabs>

When enabled, the AI can break down complex tasks into multiple steps and execute them sequentially.

#### Maximum Steps

The maximum number of steps allowed in a single plan.

**Default:** `10`  
**Range:** 1-20

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > User Experience > Planning Configuration** and adjust the "Maximum Steps" input.
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      planning: {
        enabled: true,
        maxSteps: 15
      }
    })
    ```
  </Tab>
  <Tab title="Runtime">
    ```javascript
    window.Moss.configure({
      planning: { maxSteps: 20 }
    })
    ```
  </Tab>
</Tabs>

<Info>
    Higher values allow for more complex multi-step tasks but may take longer to
    complete.
</Info>

#### Allow User Skip

Allow users to skip individual steps in a plan.

**Default:** Enabled

When enabled, users can choose to skip steps they've already completed or want to handle manually.

#### Show Plan Overview

Display the full plan overview to users before execution.

**Default:** Enabled

Shows users what the AI plans to do before taking action, improving transparency and trust.

### Session Behavior

Configure how user sessions are managed and timed out.

#### Inactivity Timeout

Time in minutes before an inactive session is considered expired.

**Default:** 30 minutes

After this period of inactivity, a new session will be created for the user.

#### Maximum Session Age

Maximum session duration in hours, regardless of activity.

**Default:** 2 hours

Sessions will automatically expire after this duration, even if the user is active.

#### Show Notifications

Display notifications when starting new sessions.

**Default:** Enabled

When enabled, users will see a notification when a new session begins.

## Security

Control access and security settings for your SDK.

### Domain Access Control

Restrict which domains can use your SDK to prevent unauthorized usage.

**Default:** Empty (all domains allowed)

<Note>**Configuration:** Dashboard only</Note>

**How it works:**

-   If the list is empty, the SDK works on any domain
-   If domains are specified, the SDK only works on those domains
-   Wildcards are not supported

**Example allowed domains:**

```
app.example.com
staging.example.com
localhost:3000
```

<Warning>
    Be careful when restricting domains. Make sure to include all environments
    where your application runs (production, staging, development).
</Warning>

### API Keys

Manage API keys for SDK authentication.

<Note>**Configuration:** Dashboard only</Note>

#### Current API Key

Your active production API key. The full key is only shown once when generated.

<Warning>
    Your API key is sensitive. Store it securely and never commit it to version
    control.
</Warning>

#### Regenerate API Key

Create a new API key and invalidate the old one.

<Danger>
    Regenerating your API key will immediately invalidate the old key. Any
    applications using the old key will stop working.
</Danger>

### Advanced SDK Configuration

Technical settings for debugging and advanced customization.

#### Screenshot Mode

Choose how screenshots are captured.

**Default:** Full Page

**Options:**

-   **Full Page:** Captures the entire scrollable page
-   **Viewport:** Captures only the visible portion of the page

<Tabs>

  </Tab>
</Tabs>

#### Log Level

Control SDK logging verbosity.

**Default:** INFO  
**Options:** ERROR, WARN, INFO, DEBUG

Higher levels include all lower levels (DEBUG includes everything).

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > Developer Options > Advanced SDK Configuration**.
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      logLevel: 'DEBUG'
    })
    ```
  </Tab>
</Tabs>

#### DOM Target Selector

CSS selector for the DOM element to observe for changes.

**Default:** `body`

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > Developer Options > Advanced SDK Configuration**.
  </Tab>
  <Tab title="SDK Init">
    ```javascript

      applicationId: 'YOUR_APP_ID',

    ```
  </Tab>
</Tabs>

<Info>

</Info>

#### Debug Mode

Enable verbose logging and debug features.

**Default:** Disabled

Useful for troubleshooting SDK integration issues.

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > Developer Options > Advanced SDK Configuration**.
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      debugMode: true
    })
    ```
  </Tab>
</Tabs>

#### DOM Stability

Configure how the SDK waits for the page to stabilize before capturing context. The SDK runs multiple detection layers in sequence (network idle, DOM mutations, browser idle, etc.) and each layer can be individually enabled or disabled.

**Default:** Network idle, DOM mutations, browser idle, and final frame layers enabled. Loading indicators and layout shift layers disabled.

<Tabs>
  <Tab title="Dashboard">
    Navigate to **Settings > Developer Options > Advanced SDK Configuration**.
  </Tab>
  <Tab title="SDK Init">
    ```javascript
    window.Moss.init({
      applicationId: 'YOUR_APP_ID',
      userId: 'USER_ID',
      stability: {
        layers: {
          loadingIndicators: true, // Enable loading indicator detection
        },
        domMutations: {
          quietPeriodMs: 1000, // Wait 1s of DOM quiet (default: 800ms)
        },
      }
    })
    ```
  </Tab>
</Tabs>

<Tip>
    See the [SDK Advanced Configuration](/en/sdk/advanced#stability-tuning) guide for full layer-by-layer documentation.
</Tip>

### Vector Store Configuration

Choose which documentation system to use for RAG (Retrieval-Augmented Generation).

<Note>**Configuration:** Dashboard only</Note>

**Default:** V1 System (for new applications)

**Options:**

-   **Legacy System:** Uses documentation_records table
-   **V1 System (New):** Uses three-tier documentation system (sources, pages, chunks)

<Info>
    V1 provides better organization with sources, pages, and chunks. It also
    preserves full page content.
</Info>

**Before switching:**

-   Ensure your documentation is ingested into the new system
-   Test RAG search results after switching
-   You can switch back anytime if needed

### Danger Zone

Irreversible actions that affect your application.

<Note>**Configuration:** Dashboard only</Note>

#### Delete Application

Permanently delete your application and all associated data.

**Status:** Coming Soon

<Danger>
    When enabled, this action will permanently remove all data including
    sessions, users, and knowledge base documents. This cannot be undone.
</Danger>

## Complete SDK Configuration Reference

### All Available Options

Here's a complete reference of all SDK configuration options:

```typescript
interface MossConfig {
    // Required
    applicationId: string; // Your application ID
    userId: string; // Unique user identifier

    // Application Features
    useVision?: boolean; // Enable vision/screenshot analysis (default: true)
    language?: "en" | "ko"; // UI language (undefined = auto-detect)

    // Recording & History
    enableSessionRecording?: boolean; // Record sessions (default: true)
    enableScreenHistoryRecording?: boolean; // Include history in AI context (default: true)

    // Planning Configuration
    planning?: {
        enabled?: boolean; // Enable planning feature (default: false)
        maxSteps?: number; // Maximum plan steps (default: 10)
        allowUserSkip?: boolean; // Allow skipping steps (default: true)
        showPlanOverview?: boolean; // Show plan before execution (default: true)
    };

    // Session Management
    sessionConfig?: {
        inactivityTimeout?: number; // Minutes before timeout (default: 30)
        maxSessionAge?: number; // Hours before expiry (default: 2)
        showNotification?: boolean; // Show session notifications (default: true)
    };

    // Technical Settings (Developer Options)
    logLevel?: "ERROR" | "WARN" | "INFO" | "DEBUG"; // Logging level (default: 'INFO')
    debugMode?: boolean; // Enable debug mode (default: false)
    screenshotMode?: "fullpage" | "viewport"; // Screenshot capture mode (default: 'fullpage')
    observeTargetSelector?: string; // DOM observation target (default: 'body')

    // DOM Stability (see SDK Advanced docs for full reference)
    stability?: {
        maxTotalWaitMs?: number; // Max total wait (default: 15000ms)
        layers?: {
            networkIdle?: boolean; // default: true
            domMutations?: boolean; // default: true
            loadingIndicators?: boolean; // default: false
            layoutShift?: boolean; // default: false
            browserIdle?: boolean; // default: true
            finalFrame?: boolean; // default: true
        };
    };
}
```

<Warning>
    **Not configurable via SDK:** AI Assistant Instructions, API Keys, Vector
    Store Configuration, Domain Access Control, and Danger Zone actions must be
    configured through the Dashboard.
</Warning>

## Configuration Best Practices

### Dashboard vs SDK Configuration

**Use Dashboard Settings When:**

-   Setting application-wide defaults
-   Configuring security policies (allowed domains, API keys)
-   Managing application identity and metadata
-   Setting AI instructions that apply to all users
-   Managing documentation/vector store configuration

**Use SDK Initialization When:**

-   Per-user customization is needed
-   Configuration depends on runtime conditions
-   Different environments need different settings
-   Integrating with existing user preferences
-   Technical debugging and development settings

**Use Runtime Configuration When:**

-   Responding to user actions (privacy mode toggle)
-   Adjusting to network conditions
-   Feature flags or A/B testing
-   Session-specific overrides

### Performance Considerations

1. **Screenshot Mode**: Use `viewport` mode if your app has very long pages to reduce processing time
2. **Stability Layers**: Disable `loadingIndicators` and `layoutShift` layers (the defaults) unless your app uses skeleton screens that reliably disappear
3. **Session Recording**: Disable for privacy-sensitive pages or users
4. **Log Level**: Use `WARN` or `ERROR` in production to reduce overhead

### Security Best Practices

1. **Never expose API keys** in client-side code
2. **Use domain restrictions** to prevent unauthorized usage
3. **Implement user consent** for session recording
4. **Sanitize userId** to avoid PII exposure
5. **Review session recordings** for sensitive data before sharing
